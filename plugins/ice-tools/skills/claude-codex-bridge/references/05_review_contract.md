# Review Contract — กติกากลางของการให้ Codex/OpenRouter ตรวจงาน (Mode B/D/E)

> **Version: V01R01 | 2026.07.10** — ONE-HOME ของ contract นี้ · ทุก agent ที่ใช้ review-mode ชี้มาที่ไฟล์นี้ ไม่เขียนกติกาซ้ำ
> **บังคับเฉพาะ review-turns** (Mode B ตรวจตามกติกา / Mode D shard / Mode E second detector) — consult/debate เบา ๆ (Mode A) ใช้ protocol เดิมใน ref 02 ไม่ต้องแบก contract นี้
> ที่มา: business-adapted จากระบบ dual-model review ของ ai-auto-work (reviewer-brief + orchestration contract) — ปรับเข้ากับงานเอกสาร/ที่ปรึกษา ไม่ใช่ CI/โค้ดล้วน

## 1. รูปแบบการรายงานปัญหา (ทุกข้อบังคับ 4 ส่วน — ห้ามพูดลอย)

```
### [Severity][RECURRING?] ID — ชื่อสั้น
- Files/Location: ไฟล์:ตำแหน่ง (slide/หน้า/section — หลายจุดคั่น comma)
- Impact: ผลกระทบที่จับต้องได้ + เงื่อนไขที่ทำให้เกิด (ห้าม "อาจมีปัญหา")
- Minimal fix: วิธีแก้เล็กสุดที่พอ ≤3 ขั้น
```
- ขาด field ใดข้อหนึ่ง = format ไม่ผ่าน — ผู้เรียกตีกลับ
- `[RECURRING]` = ปัญหาแบบเดียวกันเคยโผล่ในรอบ/งานก่อน (ดู QA log/team-memory) — ต้องพ่วง Context Repair (§8)

## 2. ระดับความร้ายแรง (นิยามฉบับงานธุรกิจ/เอกสาร)

| ระดับ | นิยาม | ตัวอย่าง |
|---|---|---|
| **Critical** | ส่งออกไปแล้วเกิดความเสียหายจริง | ข้อเท็จจริง/ตัวเลขผิด · ตอบ TOR เท็จ · เลขบวกลบไม่ foot · ชื่อบริษัท/แบรนด์/โดเมนผิด · ไฟล์เปิดไม่ได้/corrupt · citation แต่งขึ้น |
| **High** | ข้อบกพร่องที่ลูกค้า/ผู้รับมองเห็นและกระทบความน่าเชื่อถือ | ข้อมูลสำคัญตกหล่น · ความไม่สอดคล้องข้ามหน้า · font/layout พังเห็นชัด · ภาษา AI-cadence หนัก |
| **Medium** | ควรปรับแต่ไม่บล็อกการส่ง | ความอ่านง่าย · wording ปรับได้ · ความสวยงามรอง |

## 3. บรรทัดนับแต้ม + เกณฑ์ผ่าน (machine-readable — ผู้เรียกอ่านตัวเลข ไม่ตีความร้อยแก้ว)

- ทุกรายงานตรวจ**ต้องจบด้วย**: `<!-- counts: critical=X high=Y medium=Z -->`
- **เกณฑ์ผ่าน (converge) default: `critical=0 && high<=2`** — ผู้เรียกตั้งเข้มขึ้นได้ (เช่น final ลูกค้า: high=0) แต่ห้ามหย่อนกว่า default
- **ทางบังคับด้วยกลไก (แนะนำเมื่อใช้ Codex CLI):** ส่ง `--schema references/verdict.schema.json` ผ่าน helper → Codex ถูกบังคับตอบ JSON ตาม schema (verified v0.137.0: `--output-schema` มีทั้ง exec และ resume)

## 4. หยุดเมื่อไม่คืบ (Stagnation — เข้าคู่กับ Circuit Breaker ของ fleet)

- **Critical ไม่ลดลง 2 รอบติด → STOP ทันที** ห้ามวนต่อ (เลข 2 = ค่าเดียวกับ Breaker L3 ของทีม — เกณฑ์เดียวทั้งระบบ)
- ตอนหยุด: สรุปสะอาด (ติดอะไร · ลองอะไรแล้ว · counts ต่อรอบ) + เสนอทางเลือก (ก/ข/ค) ให้ผู้ใช้ตัดสิน
- เพดานแข็ง: ~5 turns ต่อวง (LOOP CAP เดิม) — stagnation trip ได้ก่อนถึงเพดานเสมอ

## 5. สถานะการปิดปัญหา (Disposition — นับ "หนี้" ให้ตรงกัน)

| สถานะ | เงื่อนไข | นับเข้าเกณฑ์ผ่านไหม |
|---|---|---|
| `FIXED` | แก้แล้ว **และ verify กับไฟล์จริงแล้ว** (§6) | ไม่นับ |
| `SPLIT(งาน, เจ้าของ)` | แยกไปงานใหม่ — ต้องระบุทั้งงานปลายทางและเจ้าของ ขาดข้อใด = OPEN | ไม่นับ (เมื่อครบเงื่อนไข) |
| `ACCEPTED_RISK(ผู้อนุมัติ, วันที่, เหตุผล, เงื่อนไขทบทวน)` | **User อนุมัติได้คนเดียวเท่านั้น — AI ห้ามอนุมัติเอง** ขาด field ใด = OPEN | ไม่นับ (เมื่อครบ) |
| `POSTPONED(รอบครบกำหนด, เหตุผล)` | เลื่อน — ยังเป็นหนี้ | **ยังนับ** |
| `OPEN` | ค้าง | นับ |

## 6. "แก้แล้ว" ต้องพิสูจน์ (FIXED-claim verification)

- ใครบอกว่าแก้แล้ว (Codex, log, หรือแม้แต่ Claude เอง) → **เปิดไฟล์จริงดูจุดนั้นก่อน** ถึงเปลี่ยนสถานะเป็น FIXED
- คำอ้างที่เทียบกับไฟล์แล้วไม่ตรง = `FIXED_CLAIM_MISMATCH` → นับเป็น Critical [RECURRING]

## 7. ตรวจของเข้าก่อนตรวจงาน (Entry Validation)

ก่อนเริ่มตรวจ ผู้ตรวจต้องได้ครบ: (ก) ตัวงานจริง (ไฟล์/ข้อความเต็ม — ไม่ใช่คำเล่า) (ข) เกณฑ์/spec ที่ใช้เทียบ (ค) scope ที่ประกาศ (ตรวจอะไร รอบที่เท่าไร)
- **ไม่ครบ → ตอบ `REVIEW_BLOCKED: INPUT_INCOMPLETE — ขาด <ระบุ>` ทันที** ห้ามเดา ห้ามไปหาเอาเอง (การหาเองปิดบังช่องโหว่ฝั่งผู้ส่ง — ปัญหาจะวนซ้ำ)
- รอบ >1: ต้องได้รายการ issue รอบก่อน → ทุกข้อเดิมต้องมีสถานะชัด (FIXED/STILL-PRESENT) ห้ามหายเงียบ

## 8. ข้อเสนอซ่อมกติกา (Context Repairs — ซ่อมที่ราก ไม่ใช่ซ่อมแค่งาน)

ทุกรายงานตรวจจบด้วยหมวด `## Context Repairs` (หรือระบุ "ไม่มี"):
- เงื่อนไขต้องเสนอ: ปัญหาแบบเดียวกันเจอ ≥2 ครั้ง · หรือกติกาที่ควรมีไม่มีอยู่ในไฟล์กติกาใด
- แต่ละข้อระบุ: **ไฟล์เป้าหมาย** (skill/agent/reference ไหน) + **ทำไมเป็นปัญหาเชิงระบบ** + **ควรเพิ่ม/แก้ว่าอะไร**
- ผู้เรียก (L1) ส่งเข้า Job Learn → เสนอผู้ใช้อนุมัติก่อนแก้ไฟล์กติกาจริง

## 9. บันไดสำรองเมื่อผู้ตรวจไม่ทำตาม format (ตามลำดับ — ยอมรับตรง ๆ ว่าคุม model อื่นไม่ได้ 100%)

1. **ชั้นกลไก:** ใช้ `--schema verdict.schema.json` (Codex CLI) — format ถูกบังคับที่ API
2. **ชั้นขอใหม่:** ไม่มี trailer/JSON → ขอ re-emit 1 ครั้ง พร้อมแนบ format ที่ต้องการ
3. **ชั้นนับเอง:** ยังไม่ได้ → Claude อ่านเนื้อหา นับ counts เอง + ระบุในรายงานว่า "counts นับโดย Claude จากเนื้อหา (ผู้ตรวจไม่ส่ง trailer)" — โปร่งใสเสมอ
- OpenRouter (ไม่มี output-schema): เริ่มที่ชั้น 2 เลย

## 10. บันไดสำรองเมื่อ backend ใช้ไม่ได้ (Degradation Ladder)

```
Codex CLI ไม่พร้อม (ไม่ login / binary หาย / error ซ้ำ)
  → ชั้น 1: เสนอ OpenRouter (openrouter-bridge — ต้องมี OPENROUTER_API_KEY · คิดเงินตาม model)
  → ชั้น 2: ผู้ตรวจ model เดียวกัน (Claude session ใหม่/บทผู้ตรวจตั้งใจหาข้อผิด ตาม contract นี้เหมือนกัน)
            + แจ้งผู้ใช้ตรง ๆ: "ตรวจข้าม model ไม่ได้ — คุณภาพต่ำกว่า (blind spot ร่วมกัน)"
  → ทุกชั้น: รายงานว่าใช้ชั้นไหนใน attribution — ไม่เงียบ
```
