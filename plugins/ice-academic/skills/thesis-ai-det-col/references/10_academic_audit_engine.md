## 0. เครื่องยนต์นี้คืออะไร (อ่านก่อนใช้)

Engine นี้คือ **ระเบียบวิธีตรวจเอกสารวิชาการไทยก่อนเผยแพร่** ที่ใช้ได้กับงานวิชาการไทย **ทุกชนิด** — ดุษฎีนิพนธ์/วิทยานิพนธ์, บทความวิชาการ, บทความวิจัย, รายงานหน่วยงาน, board paper

**กฎสูงสุดของ Engine (Prime Directive):**

> Engine นี้ **ไม่รู้เกณฑ์มาตรฐานของเอกสารเอง** และ **ห้ามจำค่าเกณฑ์ไว้ในหัว** — ค่าเกณฑ์ (จำนวนหน้า/คำ, ช่วงบทคัดย่อ, ระบบอ้างอิง, ฟอนต์, สูตรนับคำ) คือ **spec ของเอกสารแต่ละชนิด** ที่ต้อง **"ดึงสด" (Resolve)** จากแหล่งความจริงของเอกสารนั้นก่อนเสมอ (ดู §1 Step 0)

Engine = **"ตัวสกัดเกณฑ์ + เทคนิคตรวจ"** (generic, reusable)
Standard = **"เกณฑ์ที่ถูกต้องของเอกสารชิ้นนั้น"** (มาจาก skill วารสาร / Template — ไม่ใช่ความจำ)

**โครงสองชั้นของทุก Phase:**

| สัญลักษณ์ | ความหมาย | คุณสมบัติ |
|---|---|---|
| **◆ TECHNIQUE** | กฎ/วิธีตรวจที่ใช้ได้ทุกเอกสาร (regex, สูตร, two-direction cross-check) | reusable · ไม่มีค่าเกณฑ์เฉพาะเอกสารฝังตาย |
| **▸ FIELD LESSON** | บทเรียน/กับดักจากเคสจริง — *ตัวอย่างเตือนภัย* | Living List · ขยายได้เมื่อเจอเคสใหม่ · **ไม่ใช่กฎตายตัว** |

> **เหตุผลที่แยกสองชั้น:** กัน Engine over-fit เคสต้นทาง (AGJ) — TECHNIQUE ใช้ได้ทุกที่ ส่วน FIELD LESSON เป็นคลังเตือนภัยที่โตได้ ไม่ใช่เงื่อนไขบังคับที่จะ false-positive กับเอกสารชนิดอื่น

**หลักการทำงานใหญ่ (จากต้นทาง):** ตรวจทีละส่วน → แก้ → **แล้วตรวจภาพรวมซ้ำทั้งฉบับอีกรอบเสมอ** เพราะการแก้รายส่วน (โดยเฉพาะการตัดย่อ) สร้างตำหนิชนิดใหม่ได้เอง (ประโยคขาด คำเชื่อมกระจุก เนื้อหาซ้ำข้ามหัวข้อ) — ดู Phase 7

## 0.5 EXECUTION CONTEXT — Engine ทำงานได้ "ทั้งมี agent fleet และไม่มี" (อ่านก่อนใช้)

Engine นี้ออกแบบให้ **self-contained** — Claude ตัวเดียวรันครบ 7 Phase ได้ด้วยตัวเอง **ไม่ต้องพึ่ง agent อื่น** ส่วน "OWNER" ที่ระบุในแต่ละ Phase เป็น **คำแนะนำเชิง role** (การจัดทีมที่เหมาะ) ไม่ใช่เงื่อนไขบังคับ — ให้ตีความตาม environment ที่กำลังรัน:

| Environment | มี agent fleet? | วิธีตีความ "OWNER" |
|---|---|---|
| **Claude Code (เครื่องผู้ใช้)** | มี (qa-master, ผู้ทรง/สมนึก) | แบ่งงานตาม OWNER จริง — สมนึกนำ Phase 2.1-2.3+6 · **ขอ** qa-master ทำ Phase 0,1,3,4,5,7 (independent checker กัน confirmation bias) |
| **Claude บน web / สภาพไม่มี fleet** | ไม่มี | **ทำเองครบทั้ง 7 Phase** — มอง "OWNER: qa-master" เป็น "หมวกที่ต้องสวมตอนทำ Phase นั้น" (สวมหมวก checker: ตรวจ citation/format/cross-check ด้วยสายตา adversarial เหมือนเป็นคนละคนกับผู้เขียน) |

> **กฎ degrade graceful:** ถ้าอ้างถึง agent ที่ไม่มีในระบบ (เช่น "ขอ qa-master") แล้วเรียกไม่ได้ → **ทำงานนั้นเองต่อทันที ห้ามหยุด ห้ามรายงานว่า fleet หาย** — ผลลัพธ์เชิงเนื้อหาต้องเท่ากันทั้งสองทาง ต่างแค่ "ใครถือปากกา"

> **สิ่งที่ degrade ไม่ได้ (คงเสมอทุก environment):** Step 0 Resolve Standard (§1), Prime Directive (ตรวจตามมาตรฐานเอกสาร ไม่ใช่ความจำ), anti-hallucination (Phase 5), และการ re-run ทั้งฉบับรอบสุดท้าย (Phase 7) — เหล่านี้เป็นแก่นของ engine ไม่ขึ้นกับว่ามี fleet หรือไม่

> **หมายเหตุ self-check บน web (ไม่มี independent checker):** เมื่อไม่มี qa-master เป็นผู้ตรวจแยก context การที่ผู้เขียน=ผู้ตรวจมี confirmation bias สูง → ชดเชยด้วยการ **อ่านผลลัพธ์จริงซ้ำอีกรอบแบบตั้งใจหาที่ผิด** (ไม่เชื่อ summary ของตัวเอง) โดยเฉพาะ Phase 4 (cross-check ตัวเลข M/M) และ Phase 3 (font/scale) ที่ bias มองข้ามง่าย

## 1. STEP 0 — RESOLVE STANDARD (ทำก่อนตรวจทุกครั้ง — HARD GATE)

**ห้ามเริ่ม Phase 1–7 ก่อนผ่าน Step 0** — Step 0 จบลงด้วย **"Standard Card"** ที่ทุก Phase ใช้เทียบ

### 1.1 ลำดับชั้นแหล่งเกณฑ์ (Source-of-Truth Hierarchy)

```
INPUT: เอกสารที่จะตรวจ + (อาจมี) คำสั่งจาก prompt

L0  PROMPT OVERRIDE  → ผู้ใช้ระบุในคำสั่ง "ใช้ Template/เกณฑ์นี้" หรือชี้ไฟล์เกณฑ์มาตรง
                       → ใช้ทันที ข้าม L1–L3 (prompt ชนะเสมอ)

L1  SKILL ตรงชนิด    → ระบุชนิดเอกสาร → หา skill เกณฑ์ตาม Registry §1.4
                       → ดึง spec จาก skill นั้น (อ่าน reference/checklist จริงของ skill)

L2  TEMPLATE FILE    → ไม่มี skill ตรง แต่ผู้ใช้ให้ไฟล์ Template / คำแนะนำผู้เขียน / author guidelines
                       → สกัดเกณฑ์จากไฟล์จริง (เปิดอ่าน — ห้ามใช้ความจำ)

L3  ASK USER         → ไม่มีทั้ง L0/L1/L2 → HALT + ถามผู้ใช้
                       → ห้ามเดาเกณฑ์ ห้าม fabricate ตัวเลข
```

> **L0 override ได้เสมอ** — ถ้า prompt บอก "เอา Template ก่อน" หรือ "ตรวจตามไฟล์นี้" ให้ตามคำสั่งนั้น แม้จะมี skill ตรงอยู่

### 1.2 OUTPUT ของ Step 0 — "Standard Card" (artifact บังคับ)

สร้างตารางเกณฑ์ 1 หน้าของเอกสารชิ้นนี้ ก่อนตรวจ — ทุกค่าต้องมี **ที่มา** (มาจาก skill ไหน / ไฟล์ Template หน้าไหน / ผู้ใช้ยืนยัน) **ห้ามมีค่าที่ "จำมาเอง"**:

```
┌─ STANDARD CARD ────────────────────────────────────────────┐
│ ชนิดเอกสาร      : <ดุษฎีนิพนธ์ มจร / บทความ AGJ / ...>       │
│ แหล่งเกณฑ์ (L?)  : <skill ... / Template ไฟล์ ... / ผู้ใช้>   │
│─────────────────────────────────────────────────────────────│
│ จำนวนหน้า        : พื้น ___ / เพดาน ___          [ที่มา: ...] │
│ จำนวนคำรวม       : ___                            [ที่มา: ...] │
│ บทคัดย่อ (คำ)    : พื้น ___ / เพดาน ___          [ที่มา: ...] │
│ คำสำคัญ          : ___ คำ                         [ที่มา: ...] │
│ ระบบอ้างอิง      : <APA นาม-ปี / APA7 / เชิงอรรถ มจร>        │
│ พระไตรปิฎก format: <(ที.ม. (ไทย) เล่ม/ข้อ/หน้า) ...>          │
│ ฟอนต์ + ขนาด     : <TH Sarabun PSK 16/14/18/20 ...>          │
│ สูตรนับคำไทย     : <คาลิเบรต ÷___ — ถ้าไม่ทราบ = [ASK]>       │
│ เรียงบรรณานุกรม  : <ไทยก่อนอังกฤษ / ตามอักษร ...>            │
│ ขั้นต่ำอ้างอิง   : ___ รายการ                                 │
└─────────────────────────────────────────────────────────────┘
```

> **ช่องที่หาเกณฑ์ไม่ได้ = `[NEEDS USER INPUT: ...]`** — visible เสมอ ห้ามเติมเอง

### 1.3 ◆ TECHNIQUE — การคาลิเบรตสูตรนับคำ (สำคัญ ห้ามใช้สูตรในหัว)

- **ภาษาไทย:** ต้องคาลิเบรตกับวิธีนับของวารสาร/ผู้ทรงจริง — **อย่าใช้สูตรค่าคงที่จากความจำ**
- **ภาษาอังกฤษ:** นับด้วย split ช่องว่างตรง ๆ — **ห้ามใช้สูตรไทยกับข้อความอังกฤษ**

▸ **FIELD LESSON (AGJ):** เคสนี้คาลิเบรตได้ "อักขระไม่นับช่องว่าง ÷ 5.72 ≈ คำ" สอบเทียบจากตัวเลขที่ผู้ทรงนับเอง — **ตัวหาร 5.72 เป็นของเคส AGJ เท่านั้น** เอกสารชนิดอื่นต้องคาลิเบรตใหม่ ใส่ลง Standard Card

### 1.4 Mini Registry — ชนิดเอกสาร → Skill เกณฑ์ (L1)

## 2. แผนที่ Ownership (ใครทำ Phase ไหน — *เงื่อนไขตาม §0.5*)

> **อ่าน §0.5 ก่อน:** "OWNER" ด้านล่างคือการจัดทีมเมื่อ **มี agent fleet** (Claude Code). เมื่อ **ไม่มี fleet** (web) → Claude **ทำเองครบทุก Phase** โดยสวม "หมวก" ของ owner นั้น ๆ. ตารางนี้บอก "ใครเหมาะถือ Phase ไหน" ไม่ใช่ "ห้ามทำถ้าไม่ใช่ owner".

Engine นี้ถูกเรียกได้ 2 ทาง (มี fleet) หรือรันเดี่ยว (ไม่มี fleet) — แต่ละ Phase มีเจ้าของที่เหมาะ กันงานซ้ำ/ตกหล่น:

| Phase | เนื้อแท้ | OWNER | หมายเหตุ |
|---|---|---|---|
| **0** Resolve Standard + Ledger + Tracker | workflow/เอกสาร | **qa-master** | — |
| **1** Section-by-section + Citation Guard | เอกสาร | **qa-master** | — |
| **2.1** Lexicon (คำต้องห้าม AI) | voice/AI | **สมนึก** | core thesis skill |
| **2.2** Pattern (ประโยค + verb–evidence) | voice/AI | **สมนึก** | core |
| **2.3** Repetition (shingle ซ้ำ) | voice/AI | **สมนึก** | ⭐ เพิ่มจากเคสจริง |
| **3** Format บน PDF (font/scale/ขนาด) | เอกสาร | **qa-master (D7)** | — |
| **4** Cross-check อ้างอิง 2 ทิศ | เอกสาร | **qa-master** | — |
| **5** Source-of-Truth Audit | anti-halluc | **qa-master** + สมนึก (anchor) | — |
| **6** Wording (neg→pos, ศัพท์เทคนิค) | voice/wording | **สมนึก** + qa (D8) | — |
| **7** Final Gate (re-run ทั้งฉบับ) | workflow | **qa-master** (ratchet) | — |

> **มี fleet (Code):** สมนึกนำ Phase 2.1–2.3 + 6 เอง · **ขอ qa-master** ทำ Phase 0,1,3,4,5,7 (independent) · qa ส่งกลับ Phase 2.3+6 ให้สมนึก (academic voice)
> **ไม่มี fleet (web):** Claude **ทำเองครบ 7 Phase** ตามลำดับ — ใช้ self-check ตั้งใจหาที่ผิด (§0.5) ชดเชยการไม่มี independent checker

## 4. Phase 1 — รีวิว/แก้ทีละส่วน (Section-by-Section) · OWNER: qa-master

กติกาประจำทุกรอบแก้:

### ◆ TECHNIQUE
1. **Citation Guard** — ก่อนและหลังแก้ทุกครั้ง สกัดรายการอ้างอิงในส่วนนั้น (regex ปี `\(25\d\d\)` + ชื่อ) เทียบเป็น **multiset ต้องเท่ากันเป๊ะ** — อ้างอิงห้ามหาย ห้ามงอกเงียบ ๆ ถ้างอกต้องจดลง Ledger
2. **ความหมายคงเดิม** — การย่อคือตัดวิธีบรรยาย ไม่ตัดสาระ: เก็บ **ตัวเลขผลวิจัยทุกตัว**, รายการนิยาม/ขั้นตอน, ศัพท์เทคนิค + วงเล็บอังกฤษครั้งแรก
3. **เป้าการย่อเป็น %** — แจ้งตัวเลขก่อน-หลัง (คำ/อักษร) ทุกครั้ง เพื่อคุมจำนวนหน้ารวมตาม Standard Card
4. **ระวังกับดักเกณฑ์ขั้นต่ำ** — บางส่วนมี "พื้น" ไม่ใช่แค่ "เพดาน" (เทียบ Standard Card ช่อง พื้น/เพดาน) → ถ้าสั่งย่อแล้วหลุดพื้น **ให้ทักท้วง + เสนอทางเลือก**

### ▸ FIELD LESSON
- (AGJ) บทคัดย่อ 300–400 คำ: สั่งย่อ 30% จะหลุดพื้น 300 — ต้องทักท้วง **(ตัวเลข 300–400 มาจาก Standard Card ของ AGJ ไม่ใช่ค่าสากล)**

> **หลังแก้ครบทุกส่วน → กลับไปรัน Phase 2–6 ทั้งฉบับอีกรอบเสมอ (Phase 7).** จุดพังบ่อยจากการตัดย่อ: **ประโยคขาด/ประธานห้อยลอย**, signpost หัวข้อหายจนรอยต่อสะดุด, เนื้อหาเดียวกันโผล่ซ้ำสองหัวข้อ

### 5.1 ระดับคำ (Lexicon) — 2.1

#### ◆ TECHNIQUE
- **คำต้องห้าม (Confirmed AI Signatures ไทย)** ต้องเป็น 0 หรือแก้ทุกจุด: เป็นที่ทราบกันดีว่า / นับเป็นสิ่งสำคัญ / อาจกล่าวได้ว่า / ดังที่กล่าวไว้ข้างต้น / ปฏิเสธไม่ได้ว่า / กล่าวโดยสรุป / ในท้ายที่สุด / ยิ่งไปกว่านั้น / สังเคราะห์ได้ว่า / จะเห็นได้ว่า / กล่าวคือ
- **คำเฝ้าระวัง (≤1 ต่อ ~500 คำ):** อย่างลึกซึ้ง, อย่างเป็นรูปธรรม, ในยุคปัจจุบัน, โดยสรุป
- **อังกฤษ:** delve, leverage, robust, comprehensive, seamless, landscape, testament ฯลฯ → แทนด้วยคำตรง
- **เทียบกับคลังตัวอย่างวารสารจริง (Voice Profile):** คำที่ **ไม่พบในคลังเลย** ให้เลี่ยง แม้ไม่อยู่ในลิสต์สากล

#### ▸ FIELD LESSON
- (AGJ) "กล่าวคือ" ไม่พบในคลังตัวอย่างวารสารเลย → เลี่ยง แม้เป็นคำไทยปกติ

### 5.2 ระดับประโยค/ย่อหน้า (Pattern) — 2.2

#### ◆ TECHNIQUE
- **ตัวเชื่อมเข้าอ้างอิงซ้ำ:** นับความถี่ "สอดคล้องกับ / เช่นเดียวกับ / ในทิศทางเดียวกัน / ขณะที่"
- **แม่แบบประโยคซ้ำ:** "เมื่อพิจารณา…จะพบว่า" ใกล้กันเกิน, โครง Topic+3 Support ทุกย่อหน้า, ตัวเปิดย่อหน้าซ้ำ (นับ 10–12 อักษรแรกของทุกย่อหน้า)
- **สลับจังหวะประโยค:** ในย่อหน้าหลักควรมีประโยคสั้นเด็ดขาด ≥1 และประโยคยาว ≥1
- **กริยารายงานต้องตรงหลักฐาน (Verb–Evidence):** "ยืนยัน/พิสูจน์" ใช้ได้เฉพาะงานเชิงประจักษ์ที่หนักแน่น · บทความแนวคิด/ทัศนะใช้ เสนอ/ชี้/อธิบาย/มองว่า · ผลสำรวจ/ทดลองใช้ พบว่า/รายงานว่า — ไล่เช็กรายตำแหน่งเทียบชนิดเอกสารต้นทาง (เชื่อมกับ `05` §10)

#### ▸ FIELD LESSON
- **คำสองความหมาย — ระวังก่อนนับ:** "สอดคล้องกับ" ที่เป็นตัวเชื่อมอ้างอิง (แก้ได้) ต่างจากที่เป็นความหมายปกติในชื่อหลักการ (ห้ามแตะ) → **แยกชนิดก่อนนับความถี่** ไม่งั้น false-positive

### 5.3 ระดับข้อความซ้ำ (Repetition) — 2.3 ⭐ เพิ่มจากเคสจริง

#### ◆ TECHNIQUE
- **ซ้ำในย่อหน้าเดียว:** สแกน shingle ~14 อักขระที่โผล่ ≥2 ครั้ง → เจอวลีบรรยายซ้ำจริงค่อยแก้
- **ซ้ำข้ามย่อหน้าติดกัน:** shingle ~16 อักขระร่วมกันของย่อหน้า i กับ i+1 → จับเคสประโยคถูกวางซ้ำสองที่ (มักเกิดหลังตัดย่อ)
- **ซ้ำข้ามหัวข้อ (ทั้งฉบับ):** สาระเดียวกันเล่าสองหัวข้อ → ตัดสินใจ "คงทั้งคู่ถ้าทำหน้าที่ต่างกัน / ยุบเหลือที่เดียวแล้วอ้างย้อน" **และจดเหตุผล**

#### ▸ FIELD LESSON
- **กรองศัพท์โดเมนก่อน:** ชื่อกระบวนการ/เทคโนโลยีที่ซ้ำได้โดยชอบ (เช่นชื่อระบบ) ต้องกรองออกก่อนนับ shingle ไม่งั้นนับ false-positive เพียบ

## 7. Phase 4 — Cross-check อ้างอิงสองชั้น (สองทิศทางเสมอ) · OWNER: qa-master

### ◆ TECHNIQUE
1. **สกัด in-text ครบทั้งสองรูปแบบ:** นาม-ปีหน้าประโยค `ชื่อ (25xx)` และวงเล็บท้าย `(ชื่อ, 25xx)` รวมอ้างพระไตรปิฎก/เอกสารปฐมภูมิ — **ตัดส่วน "เอกสารอ้างอิง" ออกจากเนื้อหาก่อนสแกน**
2. **สกัดรายการท้ายบท:** ผู้แต่งแรก + ปี ต่อเอนทรี
3. **จับคู่สองทิศ:**
   - **4.1 ใน→ท้าย:** ทุกแหล่งที่อ้างในเนื้อหาต้องมีเอนทรีท้ายบท
   - **4.2 ท้าย→ใน:** ทุกเอนทรีท้ายบทต้องถูกอ้างในเนื้อหา ≥1 ครั้ง (ห้ามมีรายการลอย)
4. **ตรวจกติการูปแบบไปพร้อมกัน** (เทียบ Standard Card): 2 ผู้แต่ง = ใส่สองชื่อ, >2 = ชื่อแรก และคณะ · ชื่อพระ in-text ไม่ใส่นามสกุลวงเล็บ แต่ท้ายบทใส่ — สม่ำเสมอทั้งเล่ม
5. **การเรียงท้ายบท** (ตาม Standard Card): ไทยก่อนอังกฤษ, เรียงอักษรไทยตามหลักพจนานุกรม (เทียบพยัญชนะก่อน สระกำกับทีหลัง)
6. **รายงานเป็นตัวเลข:** "in-text N ตำแหน่ง / M แหล่งไม่ซ้ำ ↔ ท้ายบท M รายการ" — สองทิศต้อง **M/M**

### ▸ FIELD LESSON
- **ตัวตัดช่วงถูกหลอกได้:** คำว่า "รายการเอกสารอ้างอิง" ที่โผล่ในเนื้อความหลัก อาจหลอกตัวตัดช่วงให้ตัดผิดจุด → ตรวจขอบเขตช่วงก่อนสแกน
- (AGJ) ตัวอย่างเรียงอักษร: เกศรินทร์ < โกวิทย์, จำเนียร < จิตณรงค์, น้ำทิพย์ < นิตยา (เทียบพยัญชนะก่อน)

## 8. Phase 5 — แหล่งที่มาจริง (Source-of-Truth Audit) · OWNER: qa-master + สมนึก (anchor)

### ◆ TECHNIQUE
- **จับคู่ทุกเอนทรีท้ายบทกับ "ไฟล์จริง" ในคลังอ้างอิงรายตัว** — ไม่ใช่เช็คว่าชื่อคุ้น ๆ
- **อ้างตรง (อ้างใน) เท่านั้น** — เนื้อความที่อ้างต้องเปิดอ่านจากเอกสารต้นทางได้จริง **ห้ามอ้างผ่านบทความอื่น**
- **ประโยคไหนอ่านแล้วสะดุด → เปิดไฟล์ต้นทางอ่านทวนทันที**
- **เอกสารออนไลน์:** ยืนยันผู้แต่ง/ชื่อเรื่อง/**ปี** จากหน้าผู้เผยแพร่จริง (ปีที่อนุมานจากเนื้อหา → **ติดธงไว้จนกว่าจะยืนยัน**) เก็บไฟล์ลงคลังพร้อม URL + วันสืบค้น
- **เอกสารปฐมภูมิมาตรฐาน (พระไตรปิฎก/กฎหมาย):** ตรวจพิกัดอ้าง (เล่ม/ข้อ/หน้า) ตรงฉบับที่ระบุ — ไม่จำเป็นต้องมีไฟล์ในคลัง แต่ต้องระบุฉบับชัด

### ▸ FIELD LESSON (เคยพบจริง — anti-hallucination)
- ตัวเลขที่ถูกบีบจนความหมายเกินจริง
- คำแปล boilerplate ที่ไม่มีในต้นฉบับ
- ศัพท์ชนกับความหมายเฉพาะของโดเมน เช่น "หลักประกัน" ในงานจัดซื้อ (ความหมายเฉพาะ ≠ ความหมายทั่วไป)

> **เชื่อมกับ thesis anti-hallucination (สมนึก):** Phase 5 ใช้หลัก "ห้าม fabricate citation/number" เดียวกับ Personal Anchors ของ Mode 3 — สมนึกช่วยยืนยัน anchor (ตัวเลข/พิกัดพระไตรปิฎก) ที่ผู้ใช้ให้

## 9. Phase 6 — การเลือกใช้คำ (Wording Audit) · OWNER: สมนึก + qa (D8)

### 9.1 ◆ TECHNIQUE — Negative wording ต่อ "งานที่เราอ้าง" → ปรับ (Positive Reframe)
- สแกนรูปแบบ: `แต่ไม่ / แต่ยังมิได้ / โดยมิได้ / ยังขาด / มองข้าม / ละเลย` ที่พ่วงกับชื่อผู้ถูกอ้างอิง
- เปลี่ยนกรอบจาก **"เขาไม่ได้ทำ X"** → **"เขาให้คุณูปการในขอบเขต Y แล้วเปิดพื้นที่ให้ก้าวต่อ"** — ช่องว่างวิจัยยังชัดเท่าเดิม แต่ผู้ถูกอ้างได้เครดิตแทนคำตำหนิ

▸ **FIELD LESSON:** สำคัญในวัฒนธรรมวิชาการไทย — **ผู้ทรงอาจรู้จักเจ้าของงานที่เราวิจารณ์**

### 9.2 ◆ TECHNIQUE — Negative ที่ "ต้องคง" (Truth-first) → ห้ามทำให้นุ่มจนความจริงหาย
- ข้อค้นพบเชิงลบของผู้วิจัยต้นทางเอง (การแก้ = บิดผลวิจัยเขา)
- แกนข้อโต้แย้งของบทความ (เช่น "เทคโนโลยีอย่างเดียวขจัดทุจริตไม่หมด" — ตัดแล้วเหตุผลของกรอบจริยธรรมพัง)
- มิติวิพากษ์/ข้อจำกัดที่ผู้ทรงสั่งเพิ่ม + คำวิจารณ์กรอบของตัวเอง (จุดสร้างความน่าเชื่อถือ)
- caveat ความซื่อตรงต่อหลักฐาน (กรณีศึกษาเดียว, เงื่อนไขหน้างาน)

> **เส้นแบ่ง 9.1 vs 9.2:** negative ต่อ *งานที่เราอ้าง* = ปรับ (เครดิตเขา) · negative ที่เป็น *สาระ/ข้อค้นพบ/ข้อจำกัดจริง* = คง (truth-first) — **เชื่อมกับ Positive Wording Discipline ของระบบ: Positive-Dominant แต่ไม่ลบความจริง**

### 9.3 ◆ TECHNIQUE — ภาษาเข้าใจง่ายกว่าศัพท์เทคนิค
- Default เล่าด้วยภาษางาน/ธุรกิจที่คนทั่วไปอ่านรู้เรื่อง ใช้ศัพท์เทคนิคเมื่อจำเป็นจริง
- **ศัพท์เฉพาะอังกฤษ: ครั้งแรกต้องมีคำไทยนำ + วงเล็บอังกฤษ** เช่น การเรียนรู้ของเครื่อง (Machine Learning) — ครั้งถัดไปใช้รูปสั้นได้ · **ห้ามวนกลับมานิยามซ้ำ ห้ามใช้ตัวย่อก่อนนิยาม** (เชื่อม `05` §8 Acronym Discipline)
- **ตรวจความสม่ำเสมอของคำแปลทั้งเล่ม** (คำเดียวกันต้องแปลเหมือนเดิมทุกที่)
