## 4. DENSITY RULE (กฎคุมความหนาแน่น)

1. **Anchor rule** — ทุก register/value-word ต้องมี **1 ใน 3 anchor** มิฉะนั้น = vague filler:
   - (a) ตัวเลข/spec จริง (4 ตำแหน่งทศนิยม / 20-20-25% / TCO 5 ปี) — แข็งสุด
   - (b) mechanism/กลไก (transparency ↔ post-paid+confirmation; comprehensive ↔ 4 module นับได้)
   - (c) proof จับต้องได้ (28 ปี / first Oracle Partner)
2. **Repeat cap** — register-word แกนเดิม **≤ 3 ครั้ง/ย่อหน้า** และ **ต้องต่างบริบท** (baseline "ตอบโจทย์" ×3). ซ้ำคำเดิม+บริบทเดิมติดกัน = filler.
3. **Section density:**
   - **Investment/Commercial** → value-prose ≈ 0 (**ห้ามใส่ register-buzzword ใน section ราคา**)
   - **Exec Summary / Why-iCE / Benefits** → density สูงสุดได้ แต่ทุก vague-value ต้องมี proof ตามใน body
   - **Company Profile / Values** → values-as-proof แทน metrics-as-proof (ไม่ต้องมีตัวเลขในส่วนแนะนำตัว)
4. **Anti-hallucination lock (H3):** ห้ามแต่ง/เปลี่ยนตัวเลข "28 ปี" และ "first Oracle Application Partner"; ห้ามเติม metric ที่ไม่มีใน TOR/scope จริง.

## 4.5 ORPHAN-DATA / FLOATING-NOTE RULE (anchor ต้องพก source ติดตัว) ⭐ V01R05

> **คนละแกนกับ §4 และ §7.4** — §4 = "value-word ต้องมี anchor" (anchor หาย) · §7.4 = "เติม anchor" · **§4.5 = anchor มีอยู่แล้วแต่ลอย ไม่มี source/bridge ผู้อ่านไม่รู้ที่มา** (anchor-without-origin). คู่ตรงข้ามที่ §4/§7.4 ไม่ครอบ.

## 6.5 TRANSLATIONESE / CALQUE CATALOG (TH calque จากอังกฤษ) ⭐ V01R04

> **แกนคนละตัวกับ §6:** §6 ลิเก = **หรูเกิน (มนุษย์ขยายเกินสาระ)** · §6.5 translationese = **แปลตรงคำต่อคำจากอังกฤษ (calque — โครง EN ติดมากับวลีไทย)**. ทั้งสองโผล่พร้อมกันได้ในย่อหน้าเดียว แต่ detect/fix คนละ trigger — §6.5 **เสริม §6 ไม่ทับ**.
> **เสริม 🟦 CORE `05a` §1 ไม่ซ้ำ:** `05a` §1 จับ translationese ระดับ **โครงประโยค** (การ+กริยาเกิน · passive ไทย · อนุประโยค ซึ่ง/ที่/อัน ซ้อน · ความ+คุณศัพท์) = register-neutral. §6.5 นี้จับ translationese ระดับ **วลี/ศัพท์ธุรกิจ (lexical calque)** — วลีที่แปล 1:1 จาก business-English idiom ที่ `05a` ไม่ครอบ. เจอโครงประโยคแปล → `05a`; เจอวลีศัพท์ธุรกิจแปลตรง → ตารางนี้.
> **กฎแก้เดียวกับ §6:** **cut→anchor-swap** — ไม่ใช่แปลคำหรูเป็นคำหรูอีกคำ แต่แลกวลี calque เป็น **capability/กลไก/fact ที่ตรวจสอบได้** มิฉะนั้นเขียนเป็นภาษาธุรกิจไทยที่อ่านลื่น (idiomatic) แทน. ห้ามเติม fact ที่ไม่มีในแหล่ง (H3).
> **เคสต้นทาง:** หลุดเข้า proposal/solution จริงใน Project CH Karnchang — งานธุรกิจที่ร่าง AI-assisted แล้ว calque ติดมาจาก EN source.

## 7. ภาษาราชการของไทย (Government Thai) ⭐ V01R03

> ย้ายมาจาก `05` §2 (V06 split) — register ราชการ-ธุรกิจ ไม่ใช่ academic. ใช้กับ TOR / e-GP / GFMIS / รัฐวิสาหกิจ / หนังสือราชการ ที่ฝั่ง business ต้องตอบ.

ในเอกสารราชการไทย AI มักผลิตภาษาที่ไม่ตรงธรรมเนียมของภาษาราชการแท้:

### 7.1 ความผิดพลาดที่พบบ่อย
- ใช้ "ตาม" แทน "อาศัยอำนาจตาม" / "โดยอาศัยอำนาจตาม"
- ใช้ "การประชุม" แทน "การประชุมหารือ" / "การประชุมร่วม"
- ละเลยการระบุตำแหน่งและหน่วยงานเต็มยศ
- ใช้ภาษาตลาด: "เน้น" "ดัน" "ปลดล็อก"
- ขาดการอ้าง "ระเบียบ/ประกาศ/หนังสือสั่งการ"

### 7.2 ภาษาราชการแท้
ใช้คำขึ้นต้น/ลงท้ายตามระเบียบสำนักนายกรัฐมนตรีว่าด้วยงานสารบรรณ พ.ศ. 2526:
- "ด้วย…" / "ตามที่…" (ขึ้นต้น)
- "จึงเรียนมาเพื่อ…" / "ขอแสดงความนับถือ" (ลงท้าย)

### 7.3 ตัวอย่างการแก้
*ฉบับ AI:* "เพื่อให้การดำเนินงานเป็นไปอย่างมีประสิทธิภาพ จึงควรพิจารณาดำเนินการตามแนวทางที่กำหนด"
*ฉบับราชการแท้:* "เพื่อให้การปฏิบัติเป็นไปตามระเบียบฯ ข้อ 7 (3) จึงขอให้หน่วยงานที่เกี่ยวข้องดำเนินการให้แล้วเสร็จภายในวันที่ 30 มิถุนายน 2569 และรายงานผลให้ทราบภายใน 7 วันทำการนับแต่วันแล้วเสร็จ"

### 7.4 Personal/Cultural Anchor — ตัวอย่างสำหรับ register ราชการ (Two-Pass Technique B instance)
> ใช้ตอน Pass 2 (Specificity) — แทนข้อความลอยด้วย anchor ราชการที่ตรวจสอบได้ (อ้างกฎหมาย/หนังสือสั่งการ/สถิติหน่วยงาน). กฎกลไกอยู่ที่ `02` §4 / `04` T1 (Personal Anchor) + T12 (Cultural/Temporal); ตัวอย่าง register ราชการเก็บที่นี่:
- "ตาม พ.ร.บ.การจัดซื้อจัดจ้าง พ.ศ. 2560 มาตรา 8 (3)…" — legal anchor (อ้างมาตราจริง)
