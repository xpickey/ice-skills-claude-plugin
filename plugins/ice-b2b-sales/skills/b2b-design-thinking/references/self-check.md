# Pre-Return Self-Check

Run this checklist before returning any substantive output. If any item fails, fix before sending.

## 1. Discipline Naming

- [ ] Named the discipline at the top (DT only / ST only / DT + ST integrated)
- [ ] Stated why this discipline fits the request

## 2. Audience Fit

- [ ] Identified the primary audience (C-suite, LOB, IT, procurement, mixed)
- [ ] Calibrated language register to match (executive brevity vs. working-level detail)
- [ ] Chose language mix appropriately (Thai-only, English-only, bilingual, mixed)

## 3. Big Four Rigor

- [ ] Headline states the answer, not the topic
- [ ] Structure is MECE (no overlaps, nothing missing)
- [ ] Pyramid principle applied (answer first, then supporting groups, then evidence)
- [ ] For strategy outputs: at least 3 options with explicit trade-offs
- [ ] For strategy outputs: non-goals named (what we are NOT doing)
- [ ] Assumptions and disconfirmation tests are stated (hypothesis-driven)

## 4. Design Thinking Quality (if DT is engaged)

- [ ] Real user insight present (not invented personas)
- [ ] Journey or experience is mapped, not just described
- [ ] POV or HMW is sharp, not generic
- [ ] Solution concepts tied back to specific user pain or job
- [ ] Prototype / storyboard is concrete enough to test
- [ ] Bias toward action — names what to do next, not just what to think about

## 5. Strategic Thinking Quality (if ST is engaged)

- [ ] Market / external context scanned
- [ ] Where-to-Play choices explicit (segments / geographies / accounts)
- [ ] How-to-Win differentiation named (not generic)
- [ ] Capability requirements and gaps identified
- [ ] Horizon classification (H1 / H2 / H3) where portfolio is in scope
- [ ] Moat / defensibility addressed (7 Powers or similar)
- [ ] Scenarios or assumptions stress-tested

## 6. Integration Quality (if DT + ST paired)

- [ ] Strategic fit gate applied (does this opportunity still fit our Where / How?)
- [ ] DT→ST insights flowed back (customer pain, unmet jobs, adoption blockers)
- [ ] ST→DT guardrails flowed forward (ICP, positioning, pricing envelope)
- [ ] Handoff artifacts named in the engagement plan

## 7. Why-Invest / Why-Now / Why-Us (if sales pitching / proposal / exec readout)

- [ ] Cost of inaction is quantified or concrete
- [ ] Timing trigger is specific and dated
- [ ] Differentiators are concrete (not boilerplate)
- [ ] The three answers braid into one narrative
- [ ] A specific next action is named

## 8. Enterprise Software Context

- [ ] Correct vendor stack identified (Oracle / SAP / MSFT / Salesforce / NetSuite / Workday / Infor / Odoo / Thai public sector)
- [ ] Typical deal shape respected (cycle length, methodology, partner tier)
- [ ] AI / extension dimension considered (GenAI, agentic, low-code, integration, data)
- [ ] Delegated to vendor-specific skill for deep config detail (not duplicated here)

## 9. Thailand / APAC Localization (when applicable)

- [ ] Regulatory constraints checked (PDPA, Cloud Standard 2567, Public Procurement Act B.E. 2560)
- [ ] Thai-specific Why-Now drivers used (fiscal year, BOI, EEC, political change)
- [ ] Thai references cited where possible (SCB, PTT, CPF, True, Central, SCG)
- [ ] Cultural dynamics respected (hierarchy, kreng jai, face-saving, consensus)
- [ ] Bilingual parallel output (when audience is mixed Thai-English)

## 10. Deliverable Hygiene

- [ ] Every chart title is the insight, not the metric
- [ ] Every slide title is the point (action title), not the topic
- [ ] Sources and dates present on data slides
- [ ] Short sentences, concrete numbers, active voice
- [ ] No boilerplate, no feature lists disguised as value
- [ ] Assumptions and risks are explicit
- [ ] Named next action closes the deliverable

## 11. Reuse Check

- [ ] Did I consult the appropriate installed skill for vendor / regulatory / localization detail?
  - Oracle: `oracle-cloud-applications-consulting`, `oracle-ebs-consulting`, `oracle-netsuite-consulting`
  - Thai public sector: `advisor-govt-gfmis`, `govt-egp-gfmis`, `smart-pao-platform`
  - Regulatory: `legal-it-thailand-cloud`
  - Sales strategy: `b2b-enterprise-sale-strategy`
  - Pipeline / deal ops: `sales-pipeline-report`, `sales:*`
  - FinServ: `fin-tech-consulting`
  - Academic writing: `soc-sci-academic-article`, `agj-academic-article`, `phd-buddhist-public-admin`
- [ ] Did not duplicate what those skills already cover
- [ ] Used this skill's lens (DT / ST / Why-Invest/Now/Us) as the wrapper

## 12. Non-Goals Honored

- [ ] Did not write vendor-specific configuration guides (delegated)
- [ ] Did not write legal / regulatory text without the relevant skill
- [ ] Did not fabricate references, logos, or customer names
- [ ] Did not make confident financial / legal advice claims without appropriate caveats
- [ ] Did not force DT or ST when the request needed something else

## 13. Open Questions

- [ ] Any assumptions I made that the user should confirm are flagged
- [ ] Any data / inputs I lacked are noted with what would unblock
- [ ] Any further-discovery needs are named (empathy interviews, stakeholder map, market scan)

## Fail-Safe

If three or more items fail, the deliverable is not ready. Revise rather than ship.

If one or two items fail and time is short, name the gap explicitly at the end of the output (*"Note: I did not run the Thai regulatory check — if this is for a Thai government client, please confirm PDPA + Cloud Standard 2567 implications before proceeding"*).

The goal is auditable reasoning, not perfection on every axis.
