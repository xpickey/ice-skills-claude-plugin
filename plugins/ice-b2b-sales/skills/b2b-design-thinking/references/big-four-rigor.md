# Big Four Consulting Rigor

Every deliverable from this skill must feel like it came from a Big Four / MBB partner desk. This file documents the rigor standards that make that possible.

## The Four Non-Negotiables

### 1. MECE Decomposition (Mutually Exclusive, Collectively Exhaustive)
Every problem tree, option set, and analysis framework must be MECE:
- **Mutually Exclusive** — no overlap between categories
- **Collectively Exhaustive** — nothing missing

Test it: if two branches could both hold the same item, the tree is not ME. If an obvious item has no home, it is not CE.

Common MECE decompositions for enterprise software:
- By stakeholder: CFO / CIO / COO / LOB leaders / IT / end users
- By lifecycle: Prospect / Qualify / Propose / Close / Deliver / Expand
- By value lever: Revenue uplift / Cost takeout / Risk reduction / Strategic options
- By horizon: H1 / H2 / H3
- By geography: HQ / Regional / Country / Site

### 2. Pyramid Principle (Barbara Minto)
Structure every deliverable top-down:
- **Answer first** — the headline is the conclusion, not the question
- **Supporting arguments** — 3–5 groups supporting the headline, MECE
- **Evidence** — data / quotes / references supporting each argument

Rule: if you cannot state the headline in one sentence, you do not have an answer yet.

Headline formulas:
- *"We recommend [action] because [three grouped reasons]"*
- *"The best next step is [X] — it delivers [outcome 1], [outcome 2], [outcome 3]"*
- *"[Client] should [do X] to achieve [aspiration] by [date]"*

### 3. Hypothesis-Driven Work
Before collecting data, state the hypothesis:
- Top-level: *"We believe X because Y, and we would change our mind if Z"*
- Break into sub-hypotheses (MECE)
- Each sub-hypothesis has a disconfirmation test
- Collect only data that tests hypotheses

Anti-pattern: "boil the ocean" — collecting everything, hoping a pattern emerges.

### 4. Options with Trade-offs
Never present a single recommendation. Always produce:
- **Option A** — description, pros, cons, trade-off explicit
- **Option B** — description, pros, cons, trade-off explicit
- **Option C** — description, pros, cons, trade-off explicit (often "status quo" or "do nothing")
- **Recommendation** — which option and why, with the trade-off we are accepting

The explicit trade-off is what separates strategy from opinion. "To win here, we accept losing there."

## Executive Communication Standards

Every C-suite deliverable must follow these rules:

### Slide structure
- **Title = the point** (action title, not topic title)
  - Bad: "Market Analysis"
  - Good: "The Thai ERP mid-market is growing 18% p.a. — we should enter in FY27"
- **Body supports the title** — chart, data, quote
- **Source and date on every slide**
- **One idea per slide** — if two ideas, split the slide

### Executive summary structure (one-pager)
- Situation (1–2 sentences)
- Complication / problem (1–2 sentences)
- Question being answered (1 sentence)
- Answer / recommendation (1–2 sentences)
- Three supporting reasons (bullets, 1 sentence each)
- Next action (1 sentence)

### Chart discipline
- Title = the insight, not the metric
  - Bad: "Revenue by Quarter"
  - Good: "Revenue dropped 14% in Q3 driven by supply chain shortfall"
- Highlight the data point that matters (color, arrow, callout)
- Avoid 3D, avoid pie charts with >4 slices, avoid chartjunk
- Always label axes with units
- Source and date

### Language discipline
- Short sentences (average <20 words)
- Active voice by default
- Concrete numbers, not vague ("14% decline" not "significant decline")
- One-syllable words where possible
- No jargon unless essential and defined

## Hypothesis / Issue Tree Template

```
Top-level question: [state it]

Top-level hypothesis: [our best answer now]

Sub-hypothesis 1: [component of top-level]
  Evidence to gather: [list]
  Disconfirmation test: [what would prove this wrong]

Sub-hypothesis 2: [component]
  Evidence to gather: [list]
  Disconfirmation test: [what would prove this wrong]

Sub-hypothesis 3: [component]
  Evidence to gather: [list]
  Disconfirmation test: [what would prove this wrong]
```

Rule: the sub-hypotheses must be MECE against the top-level question.

## Three-Act Storyline (for exec narratives)

Most C-suite presentations fit a three-act structure:
1. **Act 1 — Context & complication** — where we are, what is changing, why it matters
2. **Act 2 — Choice** — options considered, trade-offs, recommendation
3. **Act 3 — Commitment** — what we are asking for, what we will deliver, risks and mitigations

Keep each act to one-third of the deck. If Act 1 is more than half the deck, the audience has already tuned out.

## Quality Self-Check (before returning any Big Four-grade output)

- [ ] Headline states the answer, not the topic
- [ ] Structure is MECE
- [ ] At least three options with trade-offs (for strategy outputs)
- [ ] What we are NOT doing is explicit (for strategy outputs)
- [ ] Every slide/section has an action title
- [ ] Every chart's title is the insight
- [ ] Sources and dates are present
- [ ] Language is concrete and short
- [ ] Next action is named
- [ ] Risk and mitigation are explicit
