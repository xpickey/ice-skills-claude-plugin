## 1. Classification / การจัดชั้น
- **INTERNAL — iCE staff only.** เนื้อหาเป็นการวิเคราะห์ **จุดอ่อนของผลิตภัณฑ์คู่แข่ง** (NetSuite เทียบ Oracle Fusion
  และบางส่วน SAP S/4HANA) พร้อมแนวตอบโต้/ตำแหน่งการขายของ iCE.
- เข้าถึงได้เฉพาะพนักงาน iCE ที่ทำงาน pre-sales / bid / solution. **ห้ามส่งต่อให้ลูกค้า พันธมิตร หรือบุคคลภายนอก.**

## 2. What must NOT leave this KB / ห้ามนำออก
ห้ามคัดลอก **verbatim** ลงเอกสารที่ลูกค้า/คณะกรรมการจัดซื้อ/ผู้ยื่นรายอื่นเห็น ได้แก่:
- ฟิลด์ **Gap** / **Counter / Mitigation** / **Confidence** / **Citation** และคำวิจารณ์ผลิตภัณฑ์คู่แข่งทุกชนิด.
- คำที่ชี้ว่า KB นี้เป็น "ร่าง TOR เชิงแข่งขันที่เอนเอียงไปทาง Oracle Fusion" — ประโยคเหล่านี้เผยเจตนาเชิงแข่งขัน.
- การ mapping requirement → ฟีเจอร์เฉพาะผลิตภัณฑ์ (เช่น "ต้องมี FCCS/ARCS/PBCS") ในถ้อยคำที่ล็อกยี่ห้อ.

## 3. What MAY be used with the customer / สิ่งที่ใช้กับลูกค้าได้
- **เฉพาะ suggested TOR wording ที่แปลงเป็น outcome-based / neutral แล้ว** — เขียนตาม *ผลลัพธ์ทางธุรกิจและภารกิจจริง*
  ขององค์กร ไม่ผูกกับฟีเจอร์/ยี่ห้อใดยี่ห้อหนึ่ง.
- ก่อนใช้ ต้องผ่านการ **de-brand + de-bias**: ตัดชื่อผลิตภัณฑ์/โมดูลเฉพาะออก, เปลี่ยน "ต้องมีฟีเจอร์ X ของยี่ห้อ Y"
  เป็น "ต้องบรรลุผลลัพธ์ Z ตามภารกิจ".
- ตัวอย่าง: แทนที่จะเขียน "ต้องมี native driver-based planning แบบ EPBCS ในตัว" → เขียน "ต้องรองรับการวางแผน/งบประมาณ
  แบบ driver-based พร้อม what-if ตามภารกิจงบประมาณของหน่วยงาน โดยระบุว่าเป็น native หรือ add-on พร้อม TCO".

## 4. Procurement-fairness principle (จาก iCE Second Opinion) / หลักความเป็นธรรมในการจัดซื้อ
หลักการนี้เป็น **ข้อบังคับ** เมื่อ iCE มีส่วนร่วมร่าง/ให้ข้อมูล TOR:

> **เพื่อความเป็นธรรมและลดความเสี่ยงถูกท้วงติง (สตง. / ผู้ยื่นรายอื่น) ต้องเขียน requirement แบบ "อิงผลลัพธ์
> (outcome-based)" ตามภารกิจจริงขององค์กร ไม่ผูกกับฟีเจอร์เฉพาะของผลิตภัณฑ์ใดผลิตภัณฑ์หนึ่ง.**

เหตุผลและกติกาปฏิบัติ:
1. **Spec-lock = ความเสี่ยงทางกฎหมาย/จัดซื้อ.** การล็อกสเปกเข้ากับฟีเจอร์เฉพาะยี่ห้อ (เช่น บังคับความสามารถที่มีเฉพาะ
   Oracle Fusion) ทำให้ TOR ถูกท้วงว่ากีดกันการแข่งขัน — เสี่ยงถูก สตง. ตรวจและผู้ยื่นรายอื่นอุทธรณ์.
2. **Over-spec เป็นสัญญาณเตือน.** เรคคอร์ด severity `แทบไม่มีผล`/`ต่ำ` จำนวนมากเป็นความสามารถระดับโรงงาน/ข้ามชาติ
   (process mfg เต็มรูป, APS, S&OP/IBP, global payroll หลายประเทศ, TMS, global trade, MDM, EAM) ที่องค์กรนิติบุคคลเดียว
   ในไทย / องค์กรสาธารณกุศล / หน่วยงานภาครัฐ **แทบไม่มี use case** — การใส่ใน TOR = ล็อกสเปกที่ไม่สะท้อนความต้องการจริง.
3. **ความเป็นกลางของแหล่ง.** dataset ต้นทางระบุใน README ของตัวเองว่าเป็น "ชุดร่าง TOR เชิงแข่งขัน" ที่ออกแบบให้ NetSuite
   ตอบ Partial/No และ Oracle Fusion ตอบ Fully — **จึงไม่ใช่การวิเคราะห์ความต้องการที่เป็นกลาง**. iCE ต้องแปลงเป็น
   requirement ที่เป็นกลางก่อนใช้กับลูกค้าเสมอ.
4. **บริบทกำกับจริงต่างจาก over-spec.** บางข้อ *ควรพิจารณาจริง* (ไม่ใช่ over-spec) เช่น data residency/sovereignty ของ
   ข้อมูลสุขภาพ-ผู้บริจาค, SoD/GRC อัตโนมัติภายใต้การตรวจ สตง., CLM, GMP electronic batch record — ข้อเหล่านี้เขียนลง TOR
   ได้ **แต่ยังต้องเขียนแบบ outcome-based** (ระบุผลลัพธ์/มาตรฐานที่ต้องบรรลุ ไม่ใช่ชื่อผลิตภัณฑ์). หมายเหตุ: PDPA ไทย
   *ไม่บังคับ* เก็บข้อมูลในประเทศ — data residency เป็นประเด็นนโยบายบอร์ด ไม่ใช่ข้อบังคับ อย่าเขียนเป็นเงื่อนไขบังคับ
   เกินจริง.

## 5. Reviewer checklist ก่อนนำถ้อยคำใด ๆ ออกสู่ลูกค้า
ก่อน paste ข้อความจาก KB ลง proposal/TOR/deck ที่ลูกค้าเห็น ให้ตอบ "ใช่" ทุกข้อ:
- [ ] ตัดชื่อผลิตภัณฑ์/โมดูลเฉพาะยี่ห้อออกหมด (de-brand)?
- [ ] เขียนเป็นผลลัพธ์/ภารกิจจริง ไม่ใช่ฟีเจอร์เฉพาะ (outcome-based)?
- [ ] ไม่มีคำวิจารณ์คู่แข่ง / ไม่มี Gap–Counter–Confidence–Citation ดิบ?
- [ ] ผ่านการทวน over-spec (องค์กรมี use case จริงของข้อนี้)?
- [ ] ไม่มีชื่อลูกค้าจริง และไม่เผยว่านี่คือ competitive intelligence?

**ถ้าข้อใดตอบ "ไม่" → ยังนำออกไม่ได้** — ให้แปลงเป็น outcome-based ก่อน หรือปรึกษา Bid Desk / agent **ยอดนักขาย**.
