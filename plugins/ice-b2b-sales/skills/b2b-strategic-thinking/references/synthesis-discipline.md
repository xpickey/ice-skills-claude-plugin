# Synthesis Discipline

Synthesis is the act of turning raw inputs (notes, transcripts, data, opinions) into a defensible *answer*. It is the single skill that separates a strategic thinker from a competent reporter.

This reference codifies the discipline. Apply it whenever the user hands over messy inputs and expects a clear recommendation.

---

## The synthesis hierarchy

Every piece of strategic output should be readable on three levels:

1. **The answer** — one sentence, falsifiable, takes a position.
2. **The reasons** — 3 to 5, MECE, parallel in form, each independently strong.
3. **The evidence** — sourced facts, quantified where possible, that prove each reason.

If any level is weak, the structure collapses. Test by deleting from the bottom up: remove the evidence — does the reason still feel sound? Remove the reasons — does the answer still convince? If yes, you've over-written. If no, you've under-supported.

---

## The five moves

Synthesis is not a single act. It is five distinct moves applied in order. Skipping a move shows up later as a hole in the recommendation.

### Move 1 — Decompose the question
Restate the user's request as a sharp question. Most input questions are too vague to answer ("what should we do about this account?"). Sharpen it: *"Should we re-pursue Account X with our Oracle ERP offering this quarter, given a 6-month stall and a competing NetSuite proposal?"*

If the question won't sharpen, name the ambiguity and ask one decisive clarifier — don't guess.

### Move 2 — Inventory the inputs
List what you actually have: transcripts, deal data, customer voice, win/loss notes, public research. Mark each input as **fact**, **claim**, or **inference**. The sources you don't have are also data — call them out as gaps.

### Move 3 — Group and find the pattern
Cluster inputs into themes. Resist the urge to use the first frame that comes to mind. Ask: *what is the surprising pattern here?* Surprise is a marker of insight. Boring synthesis (e.g., "the customer cares about price") usually means you've stopped at the obvious cluster.

### Move 4 — Stress-test each candidate insight
For each candidate insight, run three tests:
- **Falsifiability:** what evidence would prove this wrong? If "nothing," it's not an insight, it's a vibe.
- **So-what test:** if true, what changes? If "nothing changes," drop it.
- **Counter-frame:** what would a smart skeptic say? Steelman the opposite.

### Move 5 — Commit to a position
Strategy is a choice. Wishy-washy synthesis ("on the one hand… on the other…") is a failure of nerve. Take the position the evidence supports, name the confidence level, and be explicit about what would change your mind.

## Anti-patterns to watch for

These show up in your own drafts. Catch them before delivery.

| Anti-pattern | Symptom | Fix |
|---|---|---|
| **List-itis** | A bullet pile with no answer | Force a one-sentence headline first |
| **The "interesting" trap** | Insights that don't drive a decision | Apply the so-what test |
| **Framework drape** | Pasting a 2x2 over a problem that doesn't need one | Default to MECE prose |
| **False precision** | Numbers without sources | Mark every number with its provenance |
| **Sandwich-of-views** | "Here are 3 perspectives" with no recommendation | Pick one and defend it |
| **Insight inflation** | Calling a fact an insight | Insight = pattern + implication, not just observation |
| **Backfit** | Picking evidence that supports the gut answer | Inventory the disconfirming evidence too |

## The "20-minute synthesis" template

When time-boxed, use this scaffold mentally before drafting:

```
THE ANSWER (1 sentence): _______
CONFIDENCE: high / medium / low
WHY I BELIEVE THIS (3 reasons, MECE):
  R1: ___ (evidence: ___)
  R2: ___ (evidence: ___)
  R3: ___ (evidence: ___)
WHAT WOULD CHANGE MY MIND: ___
ONE OBJECTION I EXPECT: ___ → my response: ___
NEXT MOVE: ___
```

Even a 5-minute version of this beats writing prose first and hunting for the answer later.

## Synthesis under uncertainty

Most strategic situations have missing data. The discipline is not "wait for more data" — it's **make assumptions explicit, mark them, and reason despite them.**

Pattern:
- *Assuming X holds (confidence: medium), the implication is Y.*
- *If X turns out to be Z instead, the recommendation flips to W.*

This invites the user to challenge the assumption rather than the recommendation, which keeps the conversation productive.

## Calibration to Thai/APAC + Government context

A few synthesis-level adjustments matter when the operating context is Thailand, APAC, or government:

- **Surface relationships and face risks alongside economics.** A purely economic synthesis can land technically right and politically dead. Name the bunkhun, the face-loss risk, the patronage line.
- **Reframe ambiguity as patience, not weakness.** A Thai stakeholder saying "let's discuss again next month" is often a soft no. Synthesis must read these signals.
- **Government buyers prize defensibility over cleverness.** Optimize the synthesis for "this would survive an OAG/SAO audit" before "this is innovative."
- **Mid-enterprise SME owners think in cash and family legacy.** Reframe TCO as "cash out by month, cash in by month" and reference family-business multi-generational concerns.

## Self-check before sending

Before declaring a synthesis "done," ask:
1. Could a sharp executive disagree with my answer based on the evidence I cited?
2. Did I say what would change my mind?
3. Is the one-sentence headline strong enough to survive on its own?
4. Did I avoid every anti-pattern in the table above?
5. Is there a Why-Us / Why-Now / Why-Invest implication I should surface?

If any answer is "no," iterate before delivering.
